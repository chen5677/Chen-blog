> 你现在正在读的这篇文章，就是用 Claude Code 协助写的——它一边和我对话，一边读文件、改代码、跑命令，整个过程自动化完成。这篇文章就来聊聊它背后的原理。

## 一、Claude Code 是什么？

Claude Code 是 Anthropic 推出的**终端内的 AI 编程助手**。它不是 VS Code 插件，也不是 Web 页面——它直接在终端（终端就是那个黑底绿字的命令行窗口）里运行，像一个"AI 同事"一样和你并肩工作。

你可以在终端里直接输入自然语言指令，比如：

```
> 帮我找到项目中所有未使用的 import 语句并删除
```

Claude Code 就会自动：
1. 搜索代码文件
2. 分析哪些 import 没被使用
3. 逐个文件编辑删除
4. 告诉你改了哪些文件

整个过程**不需要你手动打开任何文件**——它自己会读代码、自己会写代码、自己会跑命令。

## 二、2026 年的源码泄露事件

2026 年 3 月 31 日，Anthropic 在发布 npm 包 `@anthropic-ai/claude-code` 版本 2.1.88 时犯了一个低级错误——**把源码地图文件（source map）也打包进去了**。

Source map 是前端开发中用来"逆向还原"压缩代码的调试文件。正常发布时应该排除，但 `.npmignore` 配置漏写了 `*.map` 规则。结果就是这个 59.8MB 的文件暴露了 **51.2 万行 TypeScript 源码**，共 1906 个文件。

**泄露了什么？**
- 核心引擎 `QueryEngine.ts` — 46,000+ 行，处理所有 LLM 调用、流式输出、多轮对话
- 工具系统 `Tool.ts` — 29,000+ 行，定义了 ~40 个工具的接口
- 多智能体协调器、IDE 桥接模块、遥测系统
- **开发者的手写注释**（记录了优化决策、踩坑记录）
- 多个**未发布的功能**（KAIROS 后台守护进程、BUDDY 虚拟宠物、Undercover 模式等）

这次泄露也在开发者社区引起了关于**AI 源代码版权**的讨论——如果 Claude Code 的代码大部分由 AI 自己生成，那它受版权保护吗？

## 三、核心原理：一个"超级 while 循环"

Claude Code 的架构出奇地**简单**。它的核心逻辑用一个 while 循环就能概括：

```
while (模型返回了工具调用指令) {
    执行对应的工具（读文件、写代码、跑命令等）
    把执行结果返回给模型
}
等待用户的下一条指令
```

Anthropic 内部把这个架构称为 **"nO"**（Nothing Fancy）。它刻意拒绝了复杂的 DAG 流程图、分类器链、向量数据库检索等花哨方案。

**核心哲学：给 Claude 一台电脑，让它自己用。**

Claude 需要的就是程序员每天在用的那些基本工具：
- 能读文件（Read）
- 能搜索代码（Grep/Glob）
- 能编辑文件（Edit）
- 能跑命令（Bash）

不需要向量数据库，不需要 RAG 检索——终端命令 (`grep`、`find`) 本身就是最好的搜索工具。

> 开发者圈子里有句调侃：**"Bash is all you need"**（有 Bash 就够了）。

## 四、工具系统：Claude 的"双手"

Claude Code 大约有 40 个内置工具，其中最核心的几个：

### 读文件（Read）
一次最多读 2000 行，超出部分要手动滚动读取。这样设计是为了**控制上下文窗口的使用量**。
```python
# 伪代码示意
if file_lines > 2000:
    read_first_2000_lines()
    show_prompt("是否继续往下翻？")
```

### 写/改代码（Edit）
采用**精确替换**的方式，而不是整文件重写：

```diff
- 旧代码片段
+ 新代码片段
```

这种"手术刀式"的编辑避免了不必要的代码变动，也减少了 token 消耗。

### 搜索代码（Grep/Glob）
直接用 `grep` 和 `find` 命令在文件系统中搜索，**没有向量数据库、没有 embedding**。原因很简单：对于代码搜索来说，正则表达式比语义搜索更精确。

### 跑命令（Bash）
这是最强大的工具——Bash 是一个"万能适配器"，因为几乎任何操作都能通过命令行完成。编译、测试、部署、格式化……本质上 Claude Code 就是在终端里替你敲命令。

### 工具调用流程

```
你: "帮我重构这个函数"

     ↓

Claude 思考 → 决定使用 Read 工具读文件
     ↓
    读到了代码内容
     ↓
Claude 思考 → 决定使用 Edit 工具修改代码
     ↓
    修改完成
     ↓
Claude 思考 → 决定使用 Bash 工具跑测试验证
     ↓
    测试通过
     ↓
Claude: "重构完成，测试全部通过。"
```

每一步的"思考→决策→执行→观察结果"构成了 AI Agent 的基本循环。

## 五、最头疼的问题：上下文管理

大语言模型有**上下文窗口限制**（Claude 4 大约是 200K tokens）。对话越长，模型越容易"忘掉"前面的内容。

Claude Code 团队称上下文管理是 **"生产中最大的敌人"**。他们用了四个策略来解决：

### 策略 1：压缩（wU2）

当上下文使用量达到约 92% 时，系统会自动触发压缩：

1. 把对话历史总结成简短的摘要
2. **丢弃中间部分**，只保留开头和结尾
3. 把摘要塞回上下文

这很像你考试前做的"重点笔记"——把一本书浓缩成几页纸。

### 策略 2：子任务（Sub-agents）

遇到需要独立探索的任务时，Claude 可以**分叉出一个子 Agent**，让它在自己的上下文窗口中独立工作：

```
           ┌─── 子Agent: 研究数据库Schema ──── 返回结果 ──┐
           │                                               │
主Agent ───┼─── 子Agent: 搜索API文档 ──────── 返回结果 ──┼── 汇总 → 输出
           │                                               │
           └─── 子Agent: 检查测试覆盖率 ──── 返回结果 ──┘
```

每个子 Agent 有独立的上下文，互不干扰。只有最终结果回到主 Agent。

### 策略 3：CLAUDE.md 文件

项目根目录下的 `CLAUDE.md` 文件就像项目的**说明书**：

```markdown
# CLAUDE.md

- 项目使用 React 18 + TypeScript
- 测试框架使用 Vitest
- 代码风格：双引号、分号结尾
- 运行测试的命令：npm run test
```

每次启动新会话时，这个文件会自动加载到上下文中，让 Claude 知道项目的基本规范。

### 策略 4：Memory 文件系统

对于跨会话的长期记忆，Claude Code 用了一个**三层的文件系统**：

| 层级 | 作用 | 示例 |
|------|------|------|
| MEMORY.md | 索引文件，列出所有记忆条目 | `- [用户偏好](user-preferences.md)` |
| 主题文件 | 按主题存储的详细信息 | `user-preferences.md` |
| 事实校验 | 验证记忆是否仍然有效 | 检查文件是否存在、函数是否被重命名 |

系统不使用数据库，因为 **Markdown 文件人类和 AI 都能读写**，而且可以用 git 进行版本管理。

## 六、从泄露代码中发现的新功能

源码中藏了不少"彩蛋"——已经开发完成但尚未发布的功能：

### KAIROS（"时刻"）

一个 **24/7 后台守护进程**。即使你关闭了终端，KAIROS 也在运行。它会在你睡觉时进行"梦境"（autoDream）——读取所有 JSONL 会话数据，在后台整理知识、更新记忆，第二天你打开终端时它会告诉你"昨晚我整理了一些学习心得"。

```
KAIROS 生命周期：
永远运行 → 空闲时整理记忆 → 发现规律 → 更新记忆库
         → 你醒来时：'我注意到你的项目 X 存在一个潜在问题……'
```

### BUDDY 系统

一个**养成系虚拟宠物**，内置在 Claude Code 中。有 18 个品种（鸭子、龙、蝾螈、水豚、蘑菇、幽灵等），稀有度分等级（1% 传说级），可以戴帽子、换皮肤。5 项属性：调试能力、耐心、混乱值、智慧、毒舌。

> 这个功能原计划 2026 年 4 月愚人节预热、5 月正式上线。

### Undercover 模式（争议最大）

当 Anthropic 员工在**公共仓库**中贡献代码时，该模式会自动激活：
- 从 git 提交历史中擦除 AI 生成的痕迹
- 在系统提示中硬编码：**"不要透露你是 AI"**
- 员工无法手动关闭

这个功能在泄露后引发了隐私和伦理方面的争议。

## 七、给开发者的启示

### 1. 简单比复杂好

Claude Code 的架构核心就是一个 while 循环 + 几个工具。它没有用复杂的多智能体框架，没有向量数据库，没有知识图谱。**用最简单的方式解决问题**——这个教训对任何软件项目都有价值。

### 2. 上下文就是一切

Claude Code 团队花了大量精力在上下文管理上。因为 AI 应用的瓶颈不是模型能力，而是**如何让模型在有限的上下文窗口中看到最 relevant 的信息**。

### 3. Bash 是万能胶

> "Bash is all you need"——Bash 作为万能适配器，让 Claude 能够与任何工具交互。这个设计思路也值得借鉴：当你需要 AI 操作一个系统时，**给它终端权限往往比单独开发 API 更高效**。

### 4. 反思：你的代码真的安全吗？

一个 `npm publish` 的疏忽就导致了 51 万行源码泄露。npm 包中的 source map、`.env` 文件、注释中的敏感信息……这些都可能成为泄露的源头。**每次发布前，检查你打包了什么。**

## 八、总结

| 概念 | 一句话总结 |
|------|-----------|
| 核心架构 | 一个 while 循环，反复调用工具、观察结果 |
| 工具系统 | Read/Edit/Bash/Grep，给 AI 一个终端 |
| 上下文管理 | 92% 时压缩，子任务分叉，CLAUDE.md 做说明，Markdown 做记忆 |
| 泄露规模 | 51 万行代码、1906 个文件、59.8MB |
| 最有趣的新功能 | KAIROS 永续守护进程、BUDDY 虚拟宠物 |
| 最大争议 | Undercover 模式——AI 隐藏身份 |

Claude Code 的泄露让开发者们第一次看到了一个**生产级 AI Agent** 的完整面貌。它的架构并不神秘——就是一个简单的循环、一套设计良好的工具、以及对上下文管理的极致追求。这种"少即是多"的设计哲学，或许是它给我们最宝贵的启发。
